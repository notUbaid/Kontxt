Screenshots are the highest-leverage conversion asset on your store listing — most users decide whether to download within the first 2-3 screenshots, before reading a word of your description. Treat them as marketing copy that happens to be images, not literal app captures.

## Screenshots Are Marketing, Not Documentation

A common beginner mistake is submitting raw, unedited app screenshots — they're technically accurate but convert poorly. Production-quality screenshots are designed assets: real app UI, framed with context, captions, and a deliberate narrative across the set.

> **Best Practice:** Your first screenshot does the most work. It should communicate your core value proposition in under 2 seconds — most users scroll past after the first one or two if it doesn't land.

## The Narrative Across a Screenshot Set

Don't treat each screenshot as an isolated image — design the set as a sequence that builds a case:

1. **Hero shot** — core value proposition, your best feature, immediately clear
2. **Key differentiator** — what makes you different from the obvious competitor/alternative
3. **Core workflow** — show the app actually being used, not just a static screen
4. **Secondary features** — supporting functionality that adds confidence
5. **Social proof or results** — ratings, testimonials, or outcome-focused messaging, if you have credible content for this

> **Tip:** Look at your top 3 competitors' screenshot sets before designing yours. Not to copy them, but to understand what's now considered baseline in your category — outdated-looking screenshots make an app feel outdated before anyone's even opened it.

## Technical Requirements

| Platform | Required Sizes | Notes |
|---|---|---|
| iOS | 6.7" (1290×2796px) and 6.5" (1242×2688px) iPhone minimum | Apple auto-generates other sizes from these in most cases, but verify in App Store Connect |
| iOS (iPad) | 12.9" iPad Pro size if app is universal | Required if you support iPad |
| Android | Minimum 2 screenshots, 16:9 or 9:16 phone | Tablet screenshots recommended if you support tablets |

Generate actual device-accurate screenshots from your app, not generic mockups — both Apple and Google can reject listings where screenshots don't reflect real, current app functionality.

## Adding Captions and Device Frames

Raw screenshots without context underperform. The standard high-converting pattern:

- **Device frame** — place the screenshot inside a realistic phone bezel; this is now an expected visual convention, and bare screenshots can look unfinished by comparison
- **Short caption** above or below each frame — 3-6 words, benefit-focused, not feature-labeled (e.g., "Track every workout in seconds" beats "Workout Tracking Screen")
- **Consistent background/style** across the full set — a coherent visual system reads as polished; mismatched styles per screenshot read as rushed

## Tools

| Tool | Good For |
|---|---|
| Figma | Full design control, free tier sufficient, most flexible |
| Previewed / AppLaunchpad / Shotbot | Purpose-built app screenshot generators, faster for non-designers |
| Native Xcode / Android Studio screenshot tools | Capturing accurate raw device screenshots before adding frames/captions |

Capture raw screenshots directly from a real device or accurate simulator at the required resolution, then bring them into a design tool for framing and captions — don't try to design the UI itself differently for screenshots versus what users actually see in the app.

## Common Mistakes

- Submitting raw, unframed, uncaptioned screenshots that read as unfinished
- Leading with a secondary feature instead of the core value proposition
- Inconsistent visual style across the set, making it look assembled rather than designed
- Screenshots that don't reflect current app functionality — a frequent cause of store rejection when reviewers compare against the actual binary
- Too much caption text, repeating the obvious instead of adding context
- Designing only for one device size and stretching/cropping awkwardly for others instead of generating proper sizes for each

## Localization Consideration

If you're targeting multiple markets, localize screenshot captions (and ideally the UI shown) per locale rather than submitting English captions globally — this is a meaningful conversion lever in non-English markets and increasingly expected by users.

## Using AI Here

```
Help me write screenshot captions for this app's store listing.

App purpose: [one sentence]
Core value proposition: [what makes someone download this over alternatives]
Features to highlight, in priority order: [list 4-6 features/screens]
Tone: [e.g., friendly, professional, energetic]

For each screenshot, write a 3-6 word benefit-focused caption (not a literal feature label).
Order them to build a narrative: hero value prop first, supporting features after.
```

AI is useful for caption copywriting and narrative structure; the actual screenshot design (framing, layout, visual consistency) still needs a design tool, since this is fundamentally a visual asset, not text.

> **Validation:** Show your finished screenshot set to someone unfamiliar with the app and ask them to explain, from the screenshots alone, what it does and why they'd download it. If they can't answer both in under 10 seconds, the set needs more work before submission.

## Quick Reference: Screenshot Checklist

- [ ] First screenshot communicates core value proposition immediately
- [ ] Full set tells a coherent narrative, not isolated unrelated screens
- [ ] Device-framed with consistent visual style across all images
- [ ] Captions are short, benefit-focused, not literal feature labels
- [ ] Required sizes generated for each platform/device class you support
- [ ] Screenshots reflect actual current app functionality, not outdated or aspirational UI
- [ ] Localized for any non-English markets you're targeting
