# Idea Definition

The single biggest reason hackathon teams lose is not bad code.

It is a bad idea — or a good idea that was never made concrete enough to build.

This module forces your idea into a shape that can be designed, built, and demoed in the time you have.

---

## The One-Sentence Test

Before anything else, write your idea as a single sentence using this structure:

```
[App name] helps [specific person] do [specific thing] so they can [specific outcome].
```

**Weak:**
> "An app that uses AI to help people be more productive."

**Strong:**
> "FocusBlock helps remote developers eliminate distracting app usage during deep work sessions so they can ship more code without burning out."

The weak version could describe a thousand apps. The strong version describes exactly one.

If your sentence is vague, your app will be vague. Judges cannot champion something they cannot describe.

Write your one-sentence pitch now before moving forward.

---

## The Problem Frame

A compelling hackathon idea is built on a real problem — not a feature.

Answer these three questions. Keep each answer to two sentences maximum.

**1. Who specifically has this problem?**
Not "everyone" or "people who use phones." A specific type of person in a specific situation.

**2. What is the painful moment?**
Describe the exact moment the problem occurs. What is the user doing, feeling, or failing at?

**3. What do they do today instead?**
How do people currently cope with this problem? What is wrong with the current approach?

> **Why this matters for demos:** Judges evaluate whether your app solves a real problem. If you can articulate the problem vividly in your pitch, you have already won half the argument before showing the app.

---

## Hackathon Idea Filters

Run your idea through these filters before committing to it. Answer honestly.

| Filter | Question | Red Flag |
|---|---|---|
| **Demoability** | Can a judge see the core value in under 2 minutes? | Requires setup, explanation, or imagination |
| **Buildability** | Can your team ship a working version in the time available? | Depends on hardware, approval, or complex integrations |
| **Wow factor** | Is there a moment in the demo that surprises or delights? | The app is useful but not memorable |
| **Specificity** | Is the target user concrete enough to design for? | "Everyone" or "anyone who has a phone" |
| **Novelty** | Does this feel meaningfully different from something that already exists? | A clone of an existing app with minor changes |

If your idea fails two or more filters, consider pivoting now. Pivoting in Phase 0 costs 20 minutes. Pivoting in Phase 3 costs the hackathon.

---

## The Demo Anchor

Every winning hackathon demo has a single moment that judges remember.

Define yours now.

This is the one thing a judge should still be thinking about when they go home. It might be:

- A feature that feels like magic ("it just knew what I needed")
- A visual moment that is genuinely beautiful
- A result that is surprisingly fast or accurate
- An interaction that feels unlike anything they have seen before

> **Example:** An AI app that generates a personalised workout plan is forgettable. An AI app that generates a personalised workout plan and shows your recovery risk in a beautiful animated heatmap — that is a demo anchor.

What is your demo anchor?

Write it in one sentence. If you cannot, the idea needs a stronger wow moment before you build it.

## AI Prompt — Idea Sharpening

Use this if your idea feels right but is not crisp enough yet — or if you have two ideas and cannot decide.

```
I am building a mobile app for a hackathon. I have limited time and need to
ship a demo-ready product.

My current idea(s):
[Describe your idea(s) in 2–4 sentences each]

Hackathon context:
- Theme or constraint (if any): [e.g. "must use AI", "sustainability focus", "sponsored by Stripe"]
- Team size: [number]
- Time remaining: [hours]
- Team skills: [e.g. React Native, Python, Node.js, UI/UX]

Help me:
1. Sharpen each idea into a one-sentence pitch using this format:
   "[App name] helps [specific person] do [specific thing] so they can [specific outcome]."

2. Identify the single strongest demo moment for each idea.

3. Flag any buildability risks given my team size and time.

4. If I have multiple ideas, recommend which to pursue and why.

Be direct. I need to make a decision and start building within the hour.
```

> **Evaluate the output critically:** AI will generate plausible-sounding ideas that may be technically overscoped or thematically weak. You know your team's skills and the judging criteria — apply that knowledge to filter the suggestions.

## AI Prompt — Problem Validation

Use this to stress-test whether your problem is real enough to build for.

```
I am building a hackathon mobile app and want to validate that my problem
framing is strong before I start designing.

My idea: [one-sentence pitch]
The problem I am solving: [2–3 sentences describing the painful moment]
Who has this problem: [specific user description]
What they do today instead: [current workaround]

Challenge my problem framing:
1. Is this a real, recurring problem or a rare edge case?
2. Is the target user specific enough to design a focused solution?
3. Is my "current workaround" actually good enough that users would not switch?
4. What is the strongest argument that this problem does not need a new app?
5. How would I make this problem feel urgent and visceral in a 2-minute demo?
```

---

## Idea Definition Checklist

Complete these before moving to MVP Features.

- [ ] One-sentence pitch written and passes the specificity test
- [ ] Target user is a specific person in a specific situation — not "everyone"
- [ ] The painful moment is described concretely
- [ ] The current workaround is identified
- [ ] Idea passes at least 4 of the 5 hackathon filters
- [ ] Demo anchor is defined in one sentence
- [ ] Every team member can describe the idea without referencing notes
