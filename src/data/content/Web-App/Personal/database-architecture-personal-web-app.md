# Database Architecture

Your database schema is the foundation everything else is built on. Get it wrong and you spend the rest of the project working around it. Get it right and the rest of the build feels like connecting pre-fitted pieces.

This module teaches you how to design a schema that's correct, simple, and maintainable — not perfect, but good enough to last.

---

## Think in Entities First

Before writing a single `CREATE TABLE`, think about your data in terms of **entities** — the distinct things your app tracks.

An entity is a noun in your product. Users, posts, tasks, invoices, habits, entries, projects — whatever your app is built around.

**Step 1: List your entities.**

Go back to your PRD and MVP features. Extract every noun that your app stores or displays. These become your tables.

**Example — a habit tracking app:**
- User
- Habit
- HabitLog (a record of completing a habit on a specific day)

**Example — a freelance invoice app:**
- User
- Client
- Invoice
- InvoiceLineItem

Start here. Don't jump to columns yet.

---

## Relationships Between Entities

Once you have your entities, define how they relate to each other. There are three relationship types:

**One-to-Many (most common)**
One entity owns many of another. One User has many Habits. One Invoice has many LineItems.

```
User ──< Habit          (one user, many habits)
Habit ──< HabitLog      (one habit, many log entries)
```

**Many-to-Many**
Many entities relate to many of another. A Post can have many Tags; a Tag can belong to many Posts.

```
Post >──< Tag   (implemented via a join table: PostTag)
```

**One-to-One**
One entity has exactly one of another. A User has one Profile. Less common — often a sign the data belongs in the same table.

> **💡 Tip**
> Most personal project schemas are 3–6 tables with one-to-many relationships. If you're designing more than 8 tables for an MVP, you're probably modeling future features. Cut back to what the MVP actually needs.

## Designing Your Tables

For each entity, define:
- **Primary key** — always `id`, UUID or auto-increment integer
- **Foreign keys** — references to other tables (e.g., `user_id`, `habit_id`)
- **Data columns** — the actual attributes you need
- **Timestamps** — `created_at` and `updated_at` on every table, always

**Standard column naming conventions:**
- snake_case for all column names
- `id` for primary key
- `[entity]_id` for foreign keys (e.g., `user_id`, `project_id`)
- `is_[adjective]` for booleans (e.g., `is_active`, `is_deleted`)
- `[noun]_at` for timestamps (e.g., `created_at`, `completed_at`)
- `[noun]_count` for cached counters (e.g., `comment_count`)

## A Worked Example

**Habit Tracker Schema:**

```sql
-- Every user of the app
CREATE TABLE users (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email       TEXT UNIQUE NOT NULL,
  name        TEXT,
  created_at  TIMESTAMPTZ DEFAULT NOW()
);

-- A habit a user wants to track
CREATE TABLE habits (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  name        TEXT NOT NULL,
  description TEXT,
  frequency   TEXT NOT NULL DEFAULT 'daily', -- 'daily' | 'weekly'
  is_active   BOOLEAN NOT NULL DEFAULT TRUE,
  created_at  TIMESTAMPTZ DEFAULT NOW(),
  updated_at  TIMESTAMPTZ DEFAULT NOW()
);

-- A record of completing a habit on a specific date
CREATE TABLE habit_logs (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  habit_id    UUID NOT NULL REFERENCES habits(id) ON DELETE CASCADE,
  user_id     UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  logged_date DATE NOT NULL,
  note        TEXT,
  created_at  TIMESTAMPTZ DEFAULT NOW(),

  UNIQUE(habit_id, logged_date)  -- Can't log the same habit twice on one day
);
```

Notice:
- Every table has `id`, `created_at`
- Foreign keys reference parent tables with `ON DELETE CASCADE`
- A UNIQUE constraint prevents duplicate logs
- `user_id` is on `habit_logs` for fast querying without joining through habits
- Columns are only what the MVP actually needs — no speculative future fields

## Common Schema Mistakes

> **⚠️ Mistake: Storing arrays or JSON blobs as a crutch**
> `tags TEXT[]` or `metadata JSONB` feels flexible, but it makes querying hard and relationships invisible. If you find yourself reaching for an array column, ask if you need a separate table instead.

> **⚠️ Mistake: Missing foreign key constraints**
> Without `REFERENCES`, your database won't prevent orphaned records — a `habit_log` pointing to a deleted habit. Always define foreign keys explicitly.

> **⚠️ Mistake: Using string IDs you generate yourself**
> UUIDs generated by the database (`gen_random_uuid()`) are safer and more predictable than anything you generate in application code. Let the database handle it.

> **⚠️ Mistake: No `updated_at` column**
> You will want to know when a record was last modified. Add `updated_at` to every mutable table now, even if you don't use it immediately.

> **⚠️ Mistake: Designing for future scale before shipping**
> Sharding strategies, denormalization for performance, and partition tables are real engineering topics — for systems with real load. Your personal project has zero users. Design for correctness first.

## Row Level Security (If Using Supabase)

If your database is Supabase, you must enable Row Level Security (RLS) on every table that contains user data. Without it, any authenticated user can query any other user's data via the client SDK.

```sql
-- Enable RLS on the habits table
ALTER TABLE habits ENABLE ROW LEVEL SECURITY;

-- Users can only see their own habits
CREATE POLICY "Users see own habits"
  ON habits FOR SELECT
  USING (auth.uid() = user_id);

-- Users can only insert habits for themselves
CREATE POLICY "Users insert own habits"
  ON habits FOR INSERT
  WITH CHECK (auth.uid() = user_id);

-- Users can only update their own habits
CREATE POLICY "Users update own habits"
  ON habits FOR UPDATE
  USING (auth.uid() = user_id);

-- Users can only delete their own habits
CREATE POLICY "Users delete own habits"
  ON habits FOR DELETE
  USING (auth.uid() = user_id);
```

Write these policies for every table that has a `user_id`. This is not optional — it's the security model that makes Supabase's client-side queries safe.

## Validating Your Schema

Before you start building, review your schema against this checklist:

**Structure**
- [ ] Every table has a UUID primary key
- [ ] Every table has `created_at`
- [ ] Every mutable table has `updated_at`
- [ ] All foreign keys are explicitly defined with `REFERENCES`
- [ ] `ON DELETE` behavior is specified for every foreign key (CASCADE or RESTRICT — know which and why)

**Correctness**
- [ ] UNIQUE constraints exist wherever duplicate records would be a bug
- [ ] No array or JSON columns where a join table would be clearer
- [ ] Column types match the data they hold (don't store numbers as TEXT)

**Security (Supabase)**
- [ ] RLS is enabled on every table with user data
- [ ] Policies exist for SELECT, INSERT, UPDATE, DELETE on each protected table
- [ ] No table exposes data across users without an explicit policy allowing it

**Scope**
- [ ] No table exists for a feature that's in your Out of Scope list
- [ ] No speculative columns "for future use"
