# Welcome

You're about to build an internal tool.

Not a SaaS. Not a startup pitch. A tool that solves a real, specific, annoying problem — for you, or for a small group of people you know well.

This is one of the best project types to learn software engineering on, because the feedback loop is short. You know the problem. You know the users (probably just yourself). You'll know within days if the tool actually works.

> **What "Personal Mode" means here**
> No investors, no scale requirements, no team to coordinate with. You're optimizing for: learning the *why* behind engineering decisions, keeping costs near zero, and shipping something you'll actually use. Production-grade rigor is available when a topic calls for it — but never required just to check a box.

---

## What You're Actually Building

An internal tool automates or organizes a process that currently lives in someone's head, a spreadsheet, a group chat, or a series of manual steps. Examples:

- A dashboard that tracks your freelance invoices and flags overdue ones
- A tool that turns messy CSV exports into a clean weekly report
- An approval tracker for a small club or team you run
- A personal CRM for your job search or client outreach

The common thread: **a real workflow exists today, and it's inefficient.** Your job in Phase 0 isn't to write code — it's to understand that workflow well enough that the tool you eventually build actually fixes it.

---

## Why Phase 0 Exists

Most beginners (and a lot of experienced engineers using AI tools for the first time) skip straight to "build me an app that does X." This produces something that *looks* like software but doesn't fit how the work actually happens.

Phase 0 exists to prevent that. Before any architecture or code, you'll:

- Name the problem precisely
- Map how the process works today
- Identify where time or accuracy is actually being lost
- Decide what's worth automating — and what isn't
- Scope a version 1 that's small enough to finish

> **Why this matters for AI-assisted building**
> AI tools are excellent at generating code for a spec. They cannot generate a spec for a problem you haven't defined. The clarity you build in Phase 0 is what turns your AI prompts from vague ("build me a tracker") into precise ("build me a tracker with these three states and this one business rule") — which is the difference between a tool that works and one you'll rebuild twice.

---

## What This Phase Will Produce

By the end of Phase 0, you'll have:

- A one-sentence problem definition
- A map of the current workflow, step by step
- A list of specific pain points, ranked by impact
- A clear decision on what to automate first
- A scoped MVP — small enough to build in your first real work session

Nothing here is throwaway. Every one of these feeds directly into Phase 1 (Product Design), where you'll turn this understanding into screens, flows, and structure.

## How This Course Works

Each module covers one decision or one concept — not a whole feature. You'll move through phases in order:

**Phase 0** → Discovery · **Phase 1** → Design · **Phase 2** → Architecture · **Phase 3** → Development · **Phase 4** → Production Readiness · **Phase 5** → Deployment · **Phase 6** → Operations & Improvement

You don't need to know anything about the later phases yet. Each one will make more sense once you've done the one before it.

> **A note on pace**
> There's no clock here. A personal internal tool built well over two focused weekends beats one rushed in a single afternoon and abandoned. Move when you're ready for the next module.
