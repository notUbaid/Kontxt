# Welcome to Your API Hackathon

You're not here to build a finished product. You're here to build something that **works, demos well, and proves an idea** in a fixed window of time.

This track is different from a typical hackathon track. You're building an **API product** — meaning judges, mentors, and other developers won't interact with a UI. They'll interact with **endpoints, documentation, and example requests**. That changes what "impressive" looks like.

---

## What "Hackathon Mode" Means Here

> **Tip — Optimize for proof, not polish.**
> Every module in this build favors speed and demo impact over long-term architecture. Decisions you make here might get revisited in a real production build later — that's correct, not lazy.

Across this build, you'll be pushed toward:

- The fastest stack that's still credible to experienced engineers
- An MVP scope tight enough to finish, ambitious enough to impress
- A demo experience that makes an invisible product (an API) feel tangible
- Documentation good enough that a stranger could call your API in under 5 minutes

---

## The Problem With API Products at Hackathons

Most teams building APIs lose points not on engineering — but on **legibility**. A working API nobody can see in action loses to a worse API with a slick demo.

> **Warning — An API with no demo is invisible.**
> Judges can't "see" a `POST /resources` call succeed. If you don't build a lightweight demo client, a Postman collection, or a live request console, your best engineering work goes unnoticed. Phase 2 and Phase 5 exist specifically to fix this.

---

## What Judges Actually Score

| Judges notice | Judges don't notice (in this format) |
|---|---|
| A live demo hitting real endpoints | Your database schema elegance |
| Clear, copyable example requests | Code you didn't get to show |
| Error responses that make sense | Test coverage percentage |
| A docs page that works on first try | Internal abstractions |
| One sharp use case, fully working | Five half-built features |

This is why **MVP Scope** — your next phase — matters more than feature count.

## Your Build Path

Seven phases, each producing a concrete artifact you carry forward:

1. **Discovery** — define the problem and lock MVP scope
2. **API Design** — resources, endpoints, request/response shapes
3. **Architecture** — stack, database, routing, demo app
4. **Development** — build the real thing
5. **Production Readiness** — security and performance, hackathon-appropriate
6. **Developer Experience** — Postman collection, quick start, examples
7. **Launch** — pitch, demo script, submission

Nothing here is reading for its own sake. Each module ends with a decision or artifact.
