# Pitch Deck

Your pitch deck is not the presentation. Your demo is the presentation.

The deck exists to anchor judges visually while you speak — headlines to hold attention, screenshots to show evidence, numbers to make claims concrete.

Most hackathon decks fail because they try to do too much. Slides packed with bullets, walls of text, and redundant screenshots that nobody reads. Judges look at the screen for 3 seconds between looking at you. Design for that.

---

## Slide Count

**7 slides. No more.**

| # | Slide | Time |
|---|---|---|
| 1 | Title | 10 sec |
| 2 | The Problem | 30 sec |
| 3 | The Solution | 30 sec |
| 4 | Demo | 90–120 sec |
| 5 | How It Works (Tech) | 30 sec |
| 6 | Differentiation | 30 sec |
| 7 | Team + Close | 20 sec |

Judges have seen 500-slide decks. Seven deliberate slides is a choice, not a limitation.

---

## Slide-by-Slide Breakdown

### Slide 1 — Title

```
┌────────────────────────────────┐
│                                │
│   [Store Logo / Name]          │
│   [One-line tagline]           │
│                                │
│   [Team name / Hackathon name] │
│                                │
└────────────────────────────────┘
```

- Store name large and confident
- Tagline is your value proposition sentence from Phase 0 — shortened to 8 words or fewer
- No team bios, no agenda, no "Hello we are..."

**Example tagline:** "Premium desk accessories for the focused developer."

---

### Slide 2 — The Problem

```
┌────────────────────────────────┐
│                                │
│   "Finding [product] online    │
│    is [the pain]."             │
│                                │
│   [Optional: 1 stat or quote]  │
│                                │
└────────────────────────────────┘
```

- One sentence problem statement. Large. Readable from the back of the room.
- Optional: one number that makes the problem feel real ("$X billion market, zero quality curation")
- No bullets. No list of five problems. One problem, stated clearly.

---

### Slide 3 — The Solution

```
┌────────────────────────────────┐
│                                │
│   [Store Name] is [what it is] │
│   for [who it's for].          │
│                                │
│   [Hero screenshot of store]   │
│                                │
└────────────────────────────────┘
```

- Solution statement at the top
- Your homepage or product listing screenshot fills the rest
- The screenshot should be clean, zoomed to show the most polished part of your UI
- No bullet points listing features

---

### Slide 4 — Demo

```
┌────────────────────────────────┐
│                                │
│   LIVE DEMO                    │
│                                │
│   [Optional: QR code to store] │
│                                │
└────────────────────────────────┘
```

This slide exists while you switch to the browser. It signals to judges: "this is intentional, the demo is part of the plan."

If your store is deployed, add a QR code. Judges who scan it during the presentation are highly engaged — this is a strong positive signal.

> **QR code generator:** [qr.io](https://qr.io) or [qrcode-monkey.com](https://qrcode-monkey.com) — free, no signup, download as PNG.

---

### Slide 5 — How It Works (Tech)

```
┌────────────────────────────────┐
│                                │
│   [Frontend]  →  [Backend]     │
│   React/Next    Node/Express   │
│                                │
│   [Data]      →  [Payments]    │
│   JSON/DB        Stripe        │
│                                │
└────────────────────────────────┘
```

- Simple two-by-two or linear diagram of your tech stack
- Name the actual tools: Next.js, Tailwind, Stripe, Vercel — not "a modern frontend framework"
- Judges with engineering backgrounds respect specificity
- Keep it to 4–6 components max

---

### Slide 6 — Differentiation

```
┌────────────────────────────────┐
│                                │
│   Unlike [alternative],        │
│   [Store Name] [differentiator]│
│                                │
│   [Optional: comparison table] │
│                                │
└────────────────────────────────┘
```

Simple comparison table if you want:

| | Your Store | Etsy | Amazon |
|---|---|---|---|
| Curated products |  |  |  |
| Niche focus |  |  |  |
| Branded packaging |  | Varies |  |

Keep it honest. Don't claim advantages you don't have. Judges notice.

---

### Slide 7 — Team + Close

```
┌────────────────────────────────┐
│                                │
│   [Name] — [Role]              │
│   [Name] — [Role]              │
│                                │
│   [Store URL or QR]            │
│   "Try it live →"              │
│                                │
└────────────────────────────────┘
```

- First names and roles (Frontend, Backend, Design, PM)
- Your store URL — large enough to read
- A single closing line: "We're happy to take questions."
- No "Thank you!" slide — it's filler

## Design Principles for the Deck

**Match your store's visual identity.** Use the same colors, fonts, and aesthetic as your store. A pitch deck that looks disconnected from the product creates subconscious doubt.

**One idea per slide.** If you're writing a second sentence on a slide, it probably belongs on the next slide or should be cut.

**Screenshots over mockups.** Use real screenshots of your actual store, not design tool mockups. It shows the thing works.

**Readable from 10 feet.** Body text minimum 24pt. Headlines 40pt+. If you have to squint, the font is too small.

**Dark or light — commit.** A dark-background deck with your accent color reads as modern and confident. A white-background deck with clean typography reads as professional. Don't mix.

## Use AI to Write Your Slide Copy

```
I'm creating a hackathon pitch deck for an e-commerce store. Here are my project details:

Store name: [name]
Tagline: [your value proposition, shortened]
Problem statement: [one sentence]
Solution statement: [two sentences]
Target customer: [one sentence]
Key differentiator: [one sentence]
Tech stack: [list your tools]
Team members: [names and roles]
Store URL: [if deployed]

Generate the exact copy for each of these 7 slides:
1. Title slide (tagline only — 8 words max)
2. Problem slide (one headline + optional one-line stat)
3. Solution slide (one headline + one supporting line)
4. Demo slide (one word or phrase only)
5. Tech stack slide (component labels only — no sentences)
6. Differentiation slide (one "Unlike X, we Y" sentence + 3-column comparison table)
7. Team + Close slide (closing line only — 10 words max)

Write for slides, not documents. No bullets unless the slide specifically calls for a table. Every line should be readable in under 3 seconds.
```

> **Review the output for:**
- [ ] No slide has more than 3 lines of text
- [ ] Every headline makes sense without the surrounding slide content
- [ ] The problem and solution slides could be read in the same breath
- [ ] The tech slide uses real product names, not category names

## Build the Deck

**Fastest options:**

| Tool | Why |
|---|---|
| **Canva** | Fast, templates, easy screenshot insertion |
| **Google Slides** | Real-time collaboration if you're a team |
| **Figma** | Best design control, slower to build |
| **Pitch.com** | Polished output, good for teams |

> If you're tight on time: Canva with a minimal dark template. Apply your brand colors. Drop in your screenshots. Done in 30 minutes.

**Export as PDF.** Always have a PDF backup. Keynote and PowerPoint rendering varies on unfamiliar hardware. PDF does not.

---

## Validate Before You Move On

- [ ] Deck is 7 slides or fewer
- [ ] Every slide is readable from 10 feet (minimum 24pt body text)
- [ ] Screenshots are from the real, working store — not mockups
- [ ] Deck visual style matches the store's aesthetic
- [ ] Demo slide is ready for the browser switch
- [ ] QR code tested — scans and opens the correct URL
- [ ] Tech stack names are specific and accurate
- [ ] Closing slide has the store URL
- [ ] PDF export saved and accessible offline
